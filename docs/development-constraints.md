# 开发约束

## 技术约束

- 前端保持无构建步骤，使用原生 ES Modules。
- 后端保持 CommonJS 模块风格。
- 不新增重量级框架，除非项目计划明确批准。
- 不把真实运行数据写死进代码。
- 不把上传文件、缩略图和元数据迁移到新位置，除非同步更新配置、部署和备份文档。

## 前端约束

- 所有 API 调用集中放在 `js/api.js`。
- DOM 节点引用优先放在 `js/dom.js`。
- 跨模块共享状态放在 `js/state.js`。
- 用户输入展示到 HTML 前必须转义或使用安全 DOM API。
- 新增视图状态时，需要考虑搜索、分组、标签、收藏、批量模式、上传预览之间的组合关系。
- 移动端布局不能只看桌面效果，`style.css` 中已有大量响应式规则。

## 后端约束

- 路由层负责请求校验和响应，不应绕过 `server/data/` 直接写入非规范化对象。
- JSON 文件写入使用临时文件 + rename。
- `/api` 写入请求必须经过 `server/middleware/auth.js` 的管理密码校验。
- `/api` 请求必须经过 `server/middleware/writeQueue.js` 的单进程队列；部分 GET 会补缩略图或同步元数据，因此也需要串行化。
- 文件名必须做清理，避免 Windows 保留名和非法字符。
- 删除、重命名、缩略图更新要考虑失败回滚。
- 生产环境必须设置 `DELETE_PASSWORD`，且不能使用默认值 `by-2099`。

## 数据约束

- 照片 ID 等于上传目录中的文件名。
- 标签数组需要去重和去空。
- 描述、故事名、故事描述等字段有长度限制。
- 事件日期使用 `YYYY-MM-DD` 字符串；事件名称最多 40 字。
- `contentHash` 和 `fileSize` 由后端上传流程写入，不应由前端直接伪造为可信来源。
- 分组名 `全部图片` 是系统保留名称。
- 空分组会在同步后消失。

## 文档约束

- 任何 API 变化必须更新 `docs/api.md`。
- 任何数据字段变化必须更新 `docs/data-model.md`。
- 新增上传整理字段或重复检测字段时，必须同步更新 `docs/api.md`、`docs/data-model.md` 和系统状态说明。
- 任何目录结构变化必须更新对应目录 `README.md` 和 `docs/module-map.md`。
- 任何部署方式变化必须更新 `docs/deployment-operations.md`。

## 安全约束

- 删除密码不是完整认证系统，不要把它描述成安全登录。
- 不要在前端暴露管理密钥。
- 评论和描述属于用户输入，展示时必须防止 HTML 注入。
- 上传文件必须继续限制类型和大小。
- 后续如改成账号体系，应保留只读访问和写入权限的清晰边界。

## 兼容性约束

- 当前项目支持常见现代浏览器。
- 图片格式识别范围在后端为 `jpg/jpeg/png/gif/webp`。
- 前端压缩依赖 Canvas，遇到不支持或异常时应有降级路径。
